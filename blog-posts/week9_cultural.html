How to ask better questions

When learning a new computer language, you are bound to have some questions. Luckily, there are many resources available we can use to find our answers. However, there are many things to keep in mind when asking these questions to help find our answer.

Has the question been asked already?

Stack Overflow is a great resource that has helped me countless times find the answer to the questions I have about a specific computer language. This site is supported by its hundreds of users that answer questions that others submit. However, almost every question you can think of has probably been asked already, so it is important to first search for your question before asking the community for help. Be sure to include the language you are using in the question as well as a specific question about a part of that language. For example, "Ruby array string interpolation" will return a much more focused result than "How to display an array of strings on the same line."

Be specific!

After we are sure that our question has not been asked already, it is important to be very specific when asking our question. Exactly what do we need help with? What languge are we using and what code are we working with. What is our input and our output? What results are we looking for? Here is an example of a specific question asked on Stackoverflow.com. Notice how he included the code he was working with and was very specific about the results he was trying to achieve.

Make your question approachable

A big problem I have when asking questions is phrasing them in a way to not discourage others from responding. For example, here is a question I posted to our Google+ community when I had a question about refactoring.

I am just finishing up the virus predictor challenge for week 5 and would really like some feedback about refactoring for this challenge, particularly from someone who has decent experience working with Ruby. I will copy what I wrote in my reflection: Sure, our original code had predicted_deaths and speed_of_spread both return half a sentence. Both use the same calculation, so it makes sense to combine them into one method. But is this good practice? If I learned anything from reading "Practical Object Oriented Design in Ruby," its that refactoring doesn't always just mean to make your code more DRY. I attempted to make these methods still have a single responsibility. Lets say I were to combine these two methods. What if our calculation for predicted_deaths changes? What if we write a new method that needs access to our speed_of_spead variable? If we combine these two methods we could run into much bigger problems later. Part of my goal with refactoring is to make to code easily updateable for future projects. So, I changed the methods to return the respective variables they are calculating and our virus_effects method to return that report sentence.

Whew! This is a bad question. First of all, this question is not very approachable to the people I am asking it. I start things off my saying that I want a response from someone who had alot of experience with Ruby, when really I am looking for anyone's advice! Second, this question is not very focused. I could have shortened this down to at least half of this space to make this more approachable for others to respond. It is no surprise that I only got one response to this question!
