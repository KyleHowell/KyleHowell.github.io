The benefits of test driven development

Today I will discuss the test driver approach to programming, as well as highlighting some of the benefits of this approach.

The Process

The first step in this approach is to write tests before you start coding. How do want this program to run? What results do you expect to get? These tests should pass if your program runs correctly. The more tests you write, the better. The next step is to start writing code that would make the tests pass. Once every test passes, the next step is to refactor and clean up the code. Here is a flowchart of this process.

Notice how the process starts with the tests. Then, you write just enough code to get that test to fail, then the next stop is to write code until the test passes. This is an important distinction, you are starting with a failed test instead of just jumping straight into coding.

Tests = productivity

In a 2005 study, it was discovered that programmers that write more tests tend to me more productive programmers. They studied a group of undergraduate students and compared those that used a test-driven approach to those that did not. TDD programmers were those that wrote tests before they started writing the code for the project. Overall, test-first developers tend to write more tests overall. Consequntly, more tests overall lead to higher levels of productivity.
